# SkyRoc 后端 1000 人在线并发优化方案

> 面向执行方(另一模型):本文件是可直接执行的改造清单。每一项都给出「现状 → 改成什么 → 为什么 → 验证」。请严格按顺序执行,尤其注意 **P1 的重试策略陷阱**,不理解就不要动那一项。所有注释保持中文,匹配项目既有风格。
>
> **已确认的决策(据此执行,不要再选)**:
> - **部署形态:单实例**。P2(Redis fail-fast)因此降为「建议做」而非「必做」——单实例下内存缓存降级不影响正确性,做 P2 只是为了及早暴露 Redis 故障、并为将来横向扩展留正确基线。
> - **P1 重试策略:采用方案 A**(启用 `EnableRetryOnFailure` + 改造 `UnitOfWork` 用 `IExecutionStrategy` 包裹事务)。方案 B 已作废,不要选。

## Context(为什么做这件事)

现状评估结论:当前配置下单实例难以稳定支撑 1000 人同时在线。经代码走查,主要风险集中在四处:

1. **数据库连接池未调优**:`Infrastructure/DependencyInjection.cs:25` 用 `AddDbContext`(无实例池化),`UseNpgsql` 未设 `MaxPoolSize`、无重试、无命令超时;连接串走 Neon pgbouncer,Npgsql 默认最大池 100,高并发下会排队/超时且无兜底。
2. **Redis 是每个已认证请求的硬依赖且会静默降级**:`SkyRoc/Extensions/AuthExtensions.cs` 的 `OnTokenValidated` 每请求做一次 Redis `EXISTS`;`Infrastructure/Caching/RedisServiceCollectionExtensions.cs` 在 Redis 连不上时静默回退到进程内存缓存 —— 这会让「加实例扛并发」的横向扩展彻底失效(token 跨实例不互认、登出/吊销不生效)。
3. **完全没有限流**:无 `AddRateLimiter`,突发流量无任何保护层,会直接击穿到数据库。
4. **生产环境仍开启 JSON 美化输出**:`SkyRoc/Program.cs:61` `WriteIndented = true` 恒定开启,浪费 CPU 和带宽。

目标产出:在不改变业务逻辑的前提下,让单实例能稳定承接更高并发,并**保证真正具备横向扩展能力**(即多实例部署时行为正确)。

关键前置事实(已核实):
- `ApplicationDbContext`(`Infrastructure/Data/ApplicationDbContext.cs:25`)构造函数只依赖 `DbContextOptions<ApplicationDbContext>`,**未注入其他 scoped 服务** → 可安全启用 `AddDbContextPool`。
- `UnitOfWork`(`Infrastructure/Repositories/UnitOfWork.cs`)使用**手动事务** `BeginTransactionAsync`(第 43-48 行),并提供统一入口 `ExecuteInTransactionAsync`(第 115、137 行)。**这是 P1 重试策略的关键约束点。**
- 连接串与 Redis 配置在 `SkyRoc/appsettings.json`。

## P1. 数据库:连接池 + 池化 + 重试(最高优先级)

### P1.1 修改 `Infrastructure/DependencyInjection.cs`

**现状**(第 25-26 行):
```csharp
services.AddDbContext<ApplicationDbContext>(options =>
    options.UseNpgsql(connectionString));
```

**改为**:
```csharp
services.AddDbContextPool<ApplicationDbContext>(options =>
    options.UseNpgsql(connectionString, npgsql =>
    {
        // 连接瞬断/超时自动重试,避免高并发下偶发抖动直接抛错
        npgsql.EnableRetryOnFailure(
            maxRetryCount: 3,
            maxRetryDelay: TimeSpan.FromSeconds(5),
            errorCodesToAdd: null);
        // 单条命令超时,防止慢查询长时间占用连接拖垮连接池
        npgsql.CommandTimeout(30);
    }));
```

- `AddDbContextPool` 复用 DbContext 实例,减少高并发下的分配与初始化开销。默认池大小 1024,通常够用;如需显式控制,可写 `AddDbContextPool<...>(..., poolSize: 128)`。
- **不要**改动 `UnitOfWork` 显式 `_dbContext.Dispose()` 的逻辑 —— 池化 context 的 Dispose 由 EF 内部处理为「重置并归还池」,现有代码安全。

### P1.2 ⚠️ 重试策略与手动事务的冲突(必须处理,否则事务型接口全部报错)

**背景**:一旦启用 `EnableRetryOnFailure`,EF Core 会使用 `NpgsqlRetryingExecutionStrategy`。此时**直接调用** `dbContext.Database.BeginTransactionAsync()` 会在运行时抛出:
> `The configured execution strategy 'NpgsqlRetryingExecutionStrategy' does not support user-initiated transactions.`

`UnitOfWork.BeginTransactionAsync`(第 47 行)正是手动事务,所以启用重试后,**所有走事务的业务接口都会崩**。必须二选一:

**采用方案 A(已确认):用执行策略包裹事务,保留重试能力。**
改造 `UnitOfWork` 的两个 `ExecuteInTransactionAsync`(第 115、137 行),让事务在 `IExecutionStrategy` 内运行:

```csharp
public async Task ExecuteInTransactionAsync(Func<Task> action, CancellationToken cancellationToken = default)
{
    ArgumentNullException.ThrowIfNull(action);

    var strategy = _dbContext.Database.CreateExecutionStrategy();
    await strategy.ExecuteAsync(async () =>
    {
        await BeginTransactionAsync(cancellationToken);
        try
        {
            await action();
            await CommitTransactionAsync(cancellationToken);
        }
        catch
        {
            if (HasActiveTransaction)
                await RollbackTransactionAsync(cancellationToken);
            throw;
        }
    });
}
```
泛型重载 `ExecuteInTransactionAsync<T>` 同样处理(在 `strategy.ExecuteAsync` 内返回结果)。

> ⚠️ 执行方注意:重试会**重放整个委托**,所以 `action` 内必须是幂等的、且不要在委托外部依赖只执行一次的副作用。本项目事务体基本是「读-改-存」,一般安全,但请人工确认没有「先扣库存再调外部支付」这类不可重放操作。若有,单独排除。

**同时**:全局搜索**绕过** `ExecuteInTransactionAsync`、直接调用 `IUnitOfWork.BeginTransactionAsync` 的地方:
```
grep -rn "BeginTransactionAsync" Application/ SkyRoc/
```
这些直接调用点在开启重试后同样会抛错,需改为走 `ExecuteInTransactionAsync`,或各自用 `CreateExecutionStrategy().ExecuteAsync(...)` 包裹。**逐个确认,不要漏。**

> **兜底提示**:若排查发现某个事务体确有「不可重放的外部副作用」(如先扣库存再调外部支付),不要为它硬套重试 —— 单独把那一处事务用 `dbContext.Database.CreateExecutionStrategy()` 之外的方式隔离,或对该操作关闭重试。但这是例外处理,默认全部走方案 A。

### P1.3 连接串补充池参数(改 `SkyRoc/appsettings.json` 或用环境变量覆盖)

在 `ConnectionStrings:DefaultConnection` 追加(键名对 Npgsql 大小写不敏感):
```
Maximum Pool Size=200;Timeout=15;Command Timeout=30;Connection Idle Lifetime=60;
```
- `Maximum Pool Size=200`:比默认 100 更宽,给 1000 用户留余量。**上限受 Neon/pgbouncer 实际允许的最大连接数约束**,执行方需先确认 Neon 计划的连接上限,不要盲目调高导致数据库侧拒绝连接。
- `Timeout=15`:获取连接的等待上限(秒),池满时快速失败而非无限阻塞。
- 生产环境请通过环境变量 `ConnectionStrings__DefaultConnection` 覆盖,**不要把生产连接串提交进仓库**(当前 `appsettings.json:3` 存在明文库密码,属另一安全问题,见 P5)。

## P2. 缓存/Redis:杜绝生产环境静默降级(单实例下为建议项,横向扩展的前提)

> **单实例部署下此项非阻塞**:内存缓存降级在单实例内 token 状态自洽,不影响正确性。做这一项的价值是:① 生产环境 Redis 配错/宕机时**立即暴露**而非悄悄降级埋雷;② 为将来加实例横向扩展留下正确基线。若本期只求单实例扛住并发,可将 P2 排在最后或暂缓,但强烈建议至少加上启动日志告警。

**现状**:`Infrastructure/Caching/RedisServiceCollectionExtensions.cs:54-59`,Redis 初始化失败时静默 `RegisterMemoryCache`,仅打印一行 stderr。生产环境一旦 Redis 抖动或配置错误,会悄无声息退化成单机内存缓存,导致多实例 token 状态不互通。

**改为**:在**生产环境**下,Redis 明确启用(`Enabled=true`)却连接失败时,应**启动失败(fail-fast)**,而不是降级。修改 `catch` 分支(第 54-59 行)与「禁用/缺连接串」分支(第 61-66 行):

```csharp
catch (Exception ex)
{
    if (environment.IsProduction())
        throw new InvalidOperationException(
            "[Redis] 生产环境要求 Redis 可用,但连接失败,拒绝以内存缓存降级方式启动。", ex);

    Console.Error.WriteLine(
        $"[Redis] Initialization failed, using in-memory cache only. {ex.Message}");
    RegisterMemoryCache(services);
}
```
对「`Enabled=true` 但连接串缺失」也做同样的生产环境 fail-fast 处理。**非生产环境保留降级**,方便本地开发无 Redis 也能跑。

- 保持 `EnsureRedisAvailable`(第 71 行)的启动探测不变。
- `appsettings.json:21` 当前 Redis 连接串是 `localhost:6379`,生产部署必须通过 `Redis__ConnectionString` 环境变量覆盖为真实地址,否则 fail-fast 会挡住启动(这正是期望行为)。

> 此项本身不提升单机吞吐,而是「加实例才能扛 1000 人」的**正确性前提**。当前定为单实例部署,故非本期阻塞项,但做了对未来扩展有益。

## P3. 引入限流(突发流量兜底)

ASP.NET Core 内置 `Microsoft.AspNetCore.RateLimiting`(.NET 9 无需额外 NuGet)。

### P3.1 在 `SkyRoc/Program.cs` 注册(建议放在 `AddControllers` 附近,约第 62 行后)
```csharp
builder.Services.AddRateLimiter(options =>
{
    options.RejectionStatusCode = StatusCodes.Status429TooManyRequests;
    // 全局:按客户端 IP 分区的并发+速率保护
    options.GlobalLimiter = PartitionedRateLimiter.Create<HttpContext, string>(httpContext =>
    {
        var key = httpContext.Connection.RemoteIpAddress?.ToString() ?? "unknown";
        return RateLimitPartition.GetFixedWindowLimiter(key, _ => new FixedWindowRateLimiterOptions
        {
            PermitLimit = 100,                       // 每窗口每 IP 100 次
            Window = TimeSpan.FromSeconds(10),
            QueueProcessingOrder = QueueProcessingOrder.OldestFirst,
            QueueLimit = 0
        });
    });
});
```
需要 `using System.Threading.RateLimiting;`。

### P3.2 启用中间件(在 `SkyRoc/Program.cs` 管道中)
放在 `UseRouting()` 之后、`UseAuthentication()` 之前(约第 105 行后):
```csharp
app.UseRateLimiter();
```

> 阈值 `PermitLimit=100 / 10s / IP` 是初始保守值,执行方应按真实业务调整。若前面有反向代理/负载均衡,`RemoteIpAddress` 可能是代理 IP,需先配置 `ForwardedHeaders` 中间件解析真实客户端 IP,否则会误伤(所有流量共用一个分区)。此项请在方案落地后按部署拓扑复核。
