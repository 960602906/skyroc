# P4-09 评审问题修复设计

## 目标

修复 P4-09 代码评审确认的审计安全、查询契约和分层规范问题，不推进 P4-10，不改变既有业务 API 的主要请求与响应结构。

## 修复范围

1. 操作审计只记录已认证且已通过授权的 `POST`、`PUT`、`PATCH`、`DELETE` API 请求；`AllowAnonymous` 登录和刷新接口不生成操作日志。
2. 查询参数脱敏覆盖密码、令牌、Secret 及常见密钥名称，例如 `key`、`apiKey`、`accessKey`、`privateKey`、`signingKey` 和 `encryptionKey`，同时避免把普通 `keyword` 误判为密钥。
3. 操作日志与登录日志增加通用内容关键字筛选。操作日志检索描述、URL、请求摘要、响应摘要、错误摘要和操作人；登录日志检索登录名、失败摘要、IP 和 User-Agent。字符串匹配不区分大小写。
4. 每次进入登录流程只记录一条登录日志。凭据错误、角色或权限加载失败、JWT 生成失败、缓存写入失败均记录安全失败摘要；成功流程记录成功。审计写入失败不覆盖原认证结果。
5. 异常处理与操作审计共享 HTTP 状态映射，确保审计摘要、`IsSuccess` 与最终客户端 HTTP 状态一致。
6. Web 层实现审计请求上下文访问器，Application 只依赖自有接口和数据模型，不直接使用 `IHttpContextAccessor`。
7. 新增系统支撑 AutoMapper Profile，将实体到响应 DTO 的机械映射迁移出应用服务。
8. 抽取操作审计与登录审计共用的文本裁剪工具。暂不拆分 `SystemSupportService`，避免在评审修复中扩大重构范围。

## 数据流

- 操作请求：认证 → 授权 → 操作审计中间件 → 控制器；中间件仅在当前身份已认证时启用，并在独立 DI 作用域写入日志。
- 登录请求：控制器进入认证服务 → 认证流程以单一 `finally` 审计出口记录结果 → 登录审计服务通过 Application 抽象读取 Web 层提供的 IP 与 User-Agent 摘要。
- 异常请求：应用异常映射为确定的 HTTP 状态；异常处理中间件写入该状态，操作审计使用同一映射记录结果。

## 测试策略

采用测试先行：

- 中间件测试覆盖已认证写请求、匿名写请求、密钥脱敏及普通关键字保留。
- 认证服务测试覆盖凭据失败、认证中途异常、成功和审计自身失败。
- 查询测试覆盖操作日志与登录日志内容关键字、不区分大小写和时间筛选组合。
- 异常集成测试覆盖响应状态与审计状态一致。
- AutoMapper 配置测试覆盖新增 Profile。
- 完成后运行构建、P4-09 定向测试、全量测试、格式检查和 EF 待迁移检查。

## 非目标

- 不拆分 `SystemSupportService` 为多个公开服务。
- 不改变日志表结构；新增查询条件只作用于已有字段。
- 不增加请求正文或响应正文采集。
- 不修改登录接口当前对“用户不存在”和“密码错误”的外部错误文案。
