# SkyRoc 自动业务测试与联调数据设计

## 1. 目标

在现有 401 项自动化测试基础上，建立一套可按顺序领取、执行和验收的自动测试任务清单，验证各模块真实业务规则，而不只验证接口能够返回成功。

同时建设内容充实、可长期保留的前端联调数据。普通项目数据表不得少于 20 条记录，所有业务上可填写且在当前状态下适用的字段都必须填写，不能因为字段非必填就留空。受唯一键、固定枚举或单例语义限制的表不强凑 20 条，而是填满全部合法记录。

## 2. 核心方案

采用用途隔离的双数据库：

- **自动化测试库**：仅供自动测试使用。每轮完整回归前删除并重建，应用全部迁移，然后生成确定性测试数据。测试结果必须可重复，不依赖上次运行残留数据。
- **联调演示库**：供前端长期联调和页面展示。数据生成器采用幂等写入，只补充或更新带有固定测试数据标识的记录，不清空前端已经使用的数据。

两个数据库使用不同连接串和数据库名。任何清库操作必须校验环境标识及数据库名白名单，禁止对非测试数据库执行删除、截断或重建。

## 3. 数据范围与数量

数据生成和验收以 EF Core 运行时模型与 PostgreSQL 实际目录为准，自动发现全部项目表，并相互核对，避免新增实体后任务清单漏表。`__EFMigrationsHistory` 等框架内部表不计入业务数据表。

### 3.1 默认数量基线

| 表类型 | 联调演示库目标数量 | 自动化测试库最低数量 |
| --- | ---: | ---: |
| 用户、角色、菜单、部门及基础资料 | 30–50 | 20 |
| 商品、客户、供应商、采购员、仓库、定价 | 30–80 | 20 |
| 订单、采购、库存、配送、售后、结算主表 | 50–100 | 20 |
| 业务明细、关联、审计、流水及日志表 | 100–300 | 20 |
| 固定枚举、单例配置和受唯一键限制的表 | 全部合法记录 | 全部合法记录 |

若业务链路自然生成的明细、流水或日志超过目标数量，保留自然生成结果，不为满足精确数量而删除数据。

### 3.2 字段完整性规则

1. 每条生成记录的所有可写、业务适用字段都必须赋予有意义的值，包括数据库定义为可空的联系人、电话、地址、图片、附件、备注、税号、车牌、批次、审核说明等字段。
2. 字符串不得使用空字符串、纯空白或机械占位值，如“测试”“暂无”“N/A”。名称、编码、地址和备注必须具有可辨识的业务含义。
3. 状态条件字段按业务语义填写：已审核数据必须有审核人、审核时间和意见；草稿数据不得伪造审核结果；已完成、已取消、已作废等状态必须填齐各自的操作信息。
4. 服务端生成字段、计算字段、数据库生成字段以及在当前状态下业务上明确不适用的字段允许按模型语义生成或留空，不得为了非空而制造无效状态。
5. 对每个条件字段至少生成一组使其适用的记录，保证该字段在数据集和测试中实际被覆盖。
6. 数量、价格、成本和金额使用 `Shared.Constants.NumericPrecision`，并覆盖零值、常规值、边界小数和较大值，但不得制造违反业务约束的数据。

### 3.3 数据多样性

联调数据需覆盖多个日期区间、区域、客户、商品分类、仓库、供应商和金额区间。状态型业务表至少覆盖正常主流程中的草稿、处理中、成功终态及允许保留的失败或取消状态。关联数据必须形成可追踪的完整链路，不能由互不关联的随机记录堆砌数量。

## 4. 数据生成架构

数据生成器分为四层：

1. **元数据盘点层**：读取 EF Core 模型和 PostgreSQL 表/列信息，输出表清单、列清单、主外键、唯一约束和可空性。
2. **确定性基础数据层**：按固定种子生成用户权限、商品、客户、供应商、仓库、定价和运营配置，保证重复运行得到相同业务标识。
3. **业务场景层**：优先通过应用服务或真实 HTTP API 驱动订单、采购、库存、配送、售后、财务和溯源状态流转，确保日志、流水、账单及快照由生产逻辑生成。仅对没有公开写入口的支撑数据使用受控的数据库构造器。
4. **质量验收层**：生成机器可读和 Markdown 两份报告，列出每张表的记录数、字段填充率、状态分布、孤儿外键、重复业务编码及业务一致性检查结果。

联调演示数据统一使用稳定的业务编码前缀或数据批次标识。幂等运行时按稳定键更新，不通过模糊名称匹配，也不删除无法确认归属的数据。

## 5. 自动测试分层

每个模块均从低成本验证逐步进入真实业务链路：

- **模型与约束测试**：实体配置、数据库约束、精度、唯一性、外键及迁移一致性。
- **应用服务测试**：校验计算、来源选择、状态机、幂等、并发和事务回滚。
- **真实 PostgreSQL 集成测试**：验证 EF Core InMemory 无法覆盖的事务、锁、约束、查询和精度行为。
- **HTTP 契约与授权测试**：验证认证、权限、参数校验、响应模型、错误码和 Swagger 契约。
- **端到端业务测试**：按业务文档执行跨模块主链路，并核对数据库副作用。
- **数据质量测试**：把数量、字段填充率、关联完整性及状态一致性作为全量回归的硬门禁。

## 6. 任务清单结构与顺序

正式清单将使用独立编号，从上到下每次只领取第一个未勾选任务：

1. **T0 测试基础设施与安全隔离**：双数据库配置、环境保护、迁移、重建、测试事务及报告目录。
2. **T1 全表盘点与数据质量规则**：自动发现表列、建立固定配置表例外清单、字段适用条件清单和统一验收器。
3. **T2 全量联调数据生成器**：先基础资料，再按依赖顺序生成完整业务链路，达到分级数量目标。
4. **T3 认证、个人中心与系统权限**：登录、刷新、注销、密码、角色菜单、按钮权限、越权和令牌失效。
5. **T4 商品、客户、供应商与定价**：基础资料 CRUD、关联、启停用、审核、报价和协议价优先级。
6. **T5 销售订单**：增删改查、金额计算、快照、审核状态机、非法流转、并发和幂等。
7. **T6 采购计划与采购单**：订单生成计划、合并拆分、分配、生成采购单、完成和取消。
8. **T7 入库、出库、库存和盘点**：批次、移动加权成本、锁定库存、反审核、盘盈盘亏和只追加流水。
9. **T8 配送、签收与回单**：任务生成、司机和路线、异常处理、分批配送、签收及订单状态聚合。
10. **T9 售后与取货**：退款、退货、补货、换货、审核、反审核、库存回补及重复执行保护。
11. **T10 客户账单与供应商结算**：应收应付生成、部分/全部结算、优惠、作废、余额和逆向调整。
12. **T11 溯源、报表与驾驶舱**：报告关联、二维码、统计口径、跨日期和多维聚合准确性。
13. **T12 导入导出、文件、打印与系统支撑**：文件安全、任务状态、模板字段、配置、公告和脱敏审计。
14. **T13 全链路、异常与非功能验证**：跨模块主流程、权限拒绝、事务回滚、并发、幂等和关键查询性能。
15. **T14 全量验收与前端联调交付**：全量构建测试、数据质量报告、数据库快照、联调账号和使用说明。

每个正式任务必须写明目标、前置条件、数据准备、正常场景、异常场景、数据库断言、交付物、验证命令和勾选条件。仅增加测试文件但未验证真实业务副作用，不得勾选。

## 7. 关键业务验收原则

- 主链路必须至少覆盖“销售订单 → 采购计划 → 采购单 → 采购入库 → 销售出库 → 配送 → 签收 → 客户结款”。
- 逆向链路必须覆盖售后申请、审核、取货、销售退货入库、账单冲减，以及采购退货出库、供应商应付调整。
- 审核、反审核、取消、作废、重提等操作必须验证合法状态、非法状态、重复请求和数据库副作用。
- 库存、金额、账单余额和结算余额不仅断言接口响应，还要核对批次、流水、主明细和关联记录。
- 报表与驾驶舱使用独立计算的期望值核对，不能用被测查询结果反推期望结果。
- 每个受保护接口至少覆盖允许、无权限、未认证三类结果；资源级权限需验证相邻权限不能越权。

## 8. 失败处理与可恢复性

自动化测试库失败时保留本轮日志和质量报告，但下一次仍从重建数据库开始。联调演示库生成失败时停止后续阶段，不删除已存在数据；修复后依靠稳定业务键继续幂等补齐。

固定配置表例外、条件字段例外必须集中登记，并由测试验证其理由仍与当前 EF Core 约束一致。任何新增例外都必须经过人工复核，禁止通过扩大忽略列表绕过数据质量失败。

## 9. 完成标准

只有同时满足以下条件，整套自动测试任务才算完成：

- 解决方案构建、现有测试和新增测试全部通过，代码格式无变化。
- 真实 PostgreSQL 测试库可以一条命令安全重建并完成全量回归。
- 联调演示库可以一条命令幂等生成或补齐数据。
- 所有普通项目表不少于 20 条；受约束表填满全部合法记录并有明确例外说明。
- 所有业务适用字段达到 100% 填充，条件字段在适用记录中达到 100% 填充。
- 不存在孤儿外键、非法状态组合、重复业务编码、库存负数或账实不一致。
- 输出可追踪到表、字段、模块和测试用例的验收报告。
- 更新 `docs/开发进度.md` 的当前断点、测试数量、数据库基线和最近验收日期。
