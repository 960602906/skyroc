- 依赖方向正确:`Shared ← Domain ← Application/Infrastructure ← SkyRoc`

## 一档:确认的缺陷(直接修,低风险可回退)

### 1. `UnitOfWork` 抛错类型用错

- **文件**:`Infrastructure/Repositories/UnitOfWork.cs`
- **问题**:`CommitTransactionAsync`(第 59 行)与 `RollbackTransactionAsync`(第 89 行)在"无活动事务"时抛 `InvalidCastException`,语义错误。`BeginTransactionAsync`(第 46 行)同类场景抛的是 `InvalidOperationException`。属复制粘贴 bug。
- **改法**:两处 `throw new InvalidCastException("当前没有活动事务可提交");` 改为 `throw new InvalidOperationException("当前没有活动事务可提交");`(回滚处文案改为"当前没有活动事务可回滚")。

### 2. `UnitOfWork` 释放了不属于它的 `DbContext`

- **文件**:`Infrastructure/Repositories/UnitOfWork.cs`
- **问题**:`Dispose(bool)`(第 195 行)`_dbContext?.Dispose()` 与 `DisposeAsyncCore`(第 212 行)`await _dbContext.DisposeAsync()`。`ApplicationDbContext` 由 DI 以 scoped 注册,容器负责其生命周期;`UnitOfWork` 不拥有它却主动释放,若上下文在同一请求内被其它组件复用会出问题。
- **改法**:
  - `Dispose(bool)`:删除 `_dbContext?.Dispose();`,只保留 `_transaction?.Dispose();`。
  - `DisposeAsyncCore`:删除 `await _dbContext.DisposeAsync();`,只保留对 `_transaction` 的释放。
  - 保留 `_disposed` 标志逻辑不变。

### 3. 死代码:方法级缓存整套未被调用

- **现状**:全仓 grep 无 `AddMethodCaching` / `DecorateWithCaching` / `.Decorate<` 调用点。`[Cacheable]`/`[CacheEvict]` 当前无任何运行时效果;`ICacheService` 仅被 `TokenCacheService` 直接使用。
- **涉及文件**:
  - `Application/Caching/CachingServiceCollectionExtensions.cs`
  - `Application/Caching/CachingAsyncInterceptor.cs`
  - `Shared/Common/CacheableAttribute.cs`
  - `Shared/Common/CacheEvictAttribute.cs`
- **决策点(执行前必须确认)**:
  - **方案 A(推荐)**:删除上述四个文件,减少认知负担。删除后需 build 确认无残留引用(CLAUDE.md/AGENTS.md 中的相关描述也应同步删掉)。
  - **方案 B**:保留并接入热点只读查询(如 BaseData `GetOptionsAsync`、菜单树)。此为新增功能,工作量属二档以上,应单独排期,不在本次清理内。
- **默认执行 A**,除非明确选择启用。

### 4. 过时误导注释(IsDeleted)

- **文件**:`Infrastructure/Repositories/Repository.cs:98`、`Infrastructure/Repositories/UserRepository.cs:110`
- **问题**:两处注释 `// 不需要手动过滤 IsDeleted，全局过滤器已处理 ✅`,但 `BaseEntity` 无 `IsDeleted` 字段,删除均为硬删除(`DbSet.Remove`),不存在软删除全局过滤器。注释会误导维护者。
- **改法**:删除这两行注释即可(不改任何行为)。

## 二档:重复收敛(纯重构,不改行为,需跑全量测试)

### 5. `ValidateAsync<T>` 静态助手到处复制

- **出现点**:`Application/Services/Orders/SaleOrderService.cs:267`、`Application/Services/Storage/StockInService.cs:1079`,以及其它编排服务各有一份内容相同的 `private static Task ValidateAsync<T>(IValidator<T>, T)`。
- **改法**:新建 `Application/Extensions/ValidationExtensions.cs`(命名空间 `Application.Extensions`),提供扩展方法:
  ```csharp
  public static async Task ValidateAsync<T>(this IValidator<T> validator, T instance)
  {
      var result = await validator.ValidateAsync(instance);
      if (!result.IsValid)
          throw new Application.Exceptions.ValidationException(result.Errors); // 与现有 ValidationException 构造保持一致
  }
  ```
  > 注意:先读现有各服务里 `ValidateAsync` 的实现体和 `Application.Exceptions.ValidationException` 的构造签名,照抄逻辑,不要臆造。
- 各编排服务删除私有副本,改调 `validator.ValidateAsync(dto)` 扩展形式(注意方法名冲突,必要时保留 using `Application.Extensions`)。

### 6. 审计助手三份实现

- **出现点**:`Application/Services/BaseData/BaseDataService.cs:248`(protected `ApplyCreateAudit`/`ApplyUpdateAudit`),`SaleOrderService.cs:370`、`StockInService.cs:1088`(私有重写)。三处 body 几乎一致(填 `CreateBy/CreateName` 或 `UpdateBy/UpdateName`,取自 `ICurrentUserService`)。
- **改法**:新建 `Application/Extensions/AuditExtensions.cs`,提供作用于 `BaseEntity` 的扩展:
  ```csharp
  public static void ApplyCreateAudit(this BaseEntity entity, ICurrentUserService currentUser) { ... }
  public static void ApplyUpdateAudit(this BaseEntity entity, ICurrentUserService currentUser) { ... }
  ```
  > 先读现有三处的实现,确认取的是哪些字段(userId / userName),照抄。`CreateTime`/`UpdateTime` 已在 `ApplicationDbContext.SaveChangesAsync` 自动填充,勿在此重复设置。
- 三处删除本地实现,`BaseDataService` 与两个编排服务统一调扩展。

### 7. 分页脚手架重复

- **出现点**:`Infrastructure/Repositories/Repository.cs:90-109`(基类 `GetPagedAsync`)与聚合仓储 override,如 `Infrastructure/Repositories/Orders/SaleOrderRepository.cs:35-60`。两者仅 `Include` 查询源不同,`count/order/skip/take` 逻辑每处重抄。
- **改法**:在 `Repository<T>` 增加 protected 助手:
  ```csharp
  protected async Task<(IEnumerable<T> Data, int Total)> PagedFromQueryAsync(
      IQueryable<T> source,
      Expression<Func<T, bool>>? predicate,
      int pageNumber, int pageSize,
      Expression<Func<T, object>>? orderBy = null,
      bool isDescending = false)
  { /* 现有 count/order/skip/take 逻辑 */ }
  ```
  基类 `GetPagedAsync` 用 `DbSet.AsNoTracking()` 作 source 调它;各聚合仓储 override 用其 `BuildDetailQuery()`(带 Include)作 source 调它。
- **顺带修**:`SaleOrderRepository.GetPagedAsync`(第 42-48 行)有 `predicate ?? (_ => true)` 之后又 `if (predicate != null)` 的冗余分支,统一为单一判断。

### 8. 命名空间大小写不一致

- **问题**:`Application.Interfaces`(小写 i)与 `Application.Exceptions`/`Application.Extensions`(大写)不一致,约 55 个 service 接口及其所有 using 受影响。
- **改法**:统一重命名为 `Application.Interfaces`。用 IDE 重命名或全局替换 `Application.Interfaces` → `Application.Interfaces`(含文件夹名与所有 using)。
- **风险**:遗漏 using;依赖 build 兜底。**单独一次提交**,量大易冲突,不要与其它改动混在一起。

### 9. Scrutor 版本漂移

- **问题**:`Application.csproj` 用 Scrutor 6.1.0,`Infrastructure.csproj` 用 6.0.1。
- **改法**:把 `Infrastructure/Infrastructure.csproj` 的 Scrutor 版本提到 `6.1.0`,`dotnet restore` + build 验证。

## 三档:待决策项(先拍板,本次不执行)

以下涉及方向性重构或运维/安全决策,需先确认再单独排期。

### 10. `StockInService` 有 God class 趋势

- `Application/Services/Storage/StockInService.cs`,1112 行、16 个注入依赖,同时管:单据 CRUD + 库存批次 + 移动加权成本 + 供应商账单同步。
- 内部分解良好,但责任面过宽。建议把成本/批次逻辑拆成独立应用服务(如 `IInventoryCostingService`)。需设计。

### 11. 库存↔财务硬耦合

- `StockInService` 事务内直接调 `ISupplierBillService.SyncPurchaseStockInAsync` / `RemoveBySourceDocumentAsync`(约第 311、356、400 行)。
- 可用 domain event / handler 解耦,但需评估事务边界一致性。

### 12. 可观测性缺失

- 仅默认 `Microsoft.Extensions.Logging`,无结构化日志、无 OpenTelemetry/metrics/tracing。生产排障靠自建审计表。
- 建议选型 Serilog + OpenTelemetry。需选型与接入设计。

### 13. 密钥明文入仓(安全,建议优先)

- `SkyRoc/appsettings.json` 含疑似真实的 Postgres 密码与 JWT secret;`appsettings.Development.json` 含 RustFS access/secret key 明文。与文档"应走环境变量"矛盾。
- 生产路径虽对占位符 `__SET_IN_ENV__` fail-fast,但明文入仓本身是风险。
- 建议:凭据迁出到环境变量 / user-secrets,并**轮换已泄露凭据**(密码、JWT signing key、RustFS key)。

### 14. 启动迁移/种子被注释

- `SkyRoc/Program.cs:99-105` 的 seeding / `MigrateAsync` 块被注释,意味着开机不自动迁移,与 CLAUDE.md 描述("auto-runs migrations and dev seeding on startup")不符。
- 需确认:是有意关闭(改文档),还是遗留误注释(恢复)。
