# 前端菜单更新工作量评估

> 依据：[`docs/前端菜单分布.md`](./前端菜单分布.md)  
> 核对基准：本仓库 `src/pages/(base)` + `VITE_AUTH_ROUTE_MODE=static`（2026-07-16）  
> 用途：排期与范围裁剪；**不是**强制一次做完全部业务页的实施清单。

## 1. 结论

| 范围 | 含义 | 前端工作量（约） | 后端配合 |
| --- | --- | --- | --- |
| **A. 菜单骨架落地** | 侧栏树、路由占位、i18n、动态路由联调 | **3–6 人天** | 替换 `DbSeeder` / `sys_menu` 种子 + Admin 绑菜单 |
| **B. A + 现有页迁到文档 path** | 把已有 goods/customer/purchase/storage/manage 迁到 `/master`、`/system` 等 | **再加 2–4 人天** | `component` / `name` 与前端一致 |
| **C. 全量业务页面** | 文档约 **63** 个叶子菜单都做成可用列表/表单 | **40–80+ 人天** | 各业务 API 已就绪为前提 |

文档本身写的是「分布与权限映射，不强制一次改完种子」。若「更新菜单」= **侧栏结构对齐**，落在 **A（或 A+B）**，不是 C。

一句话：

- 只改侧栏/动态菜单结构：约 **前端 3–6 人天 + 后端种子 2–4 人天**
- 顺带把现有页面 path 对齐文档：合计约 **一周内（1 人）**
- 菜单里每一页都做成真实业务：约 **数人周～数人月**，不是「改菜单」本身

## 2. 现状 vs 文档差距

### 2.1 菜单来源模式

| 模式 | 环境变量 | 侧栏来源 |
| --- | --- | --- |
| 当前 | `VITE_AUTH_ROUTE_MODE=static` | `src/pages/(base)` → `pnpm gen-route` → 已注册路由 |
| 文档目标 | `dynamic` | 后端 `sys_menu` + `GET /auth/getRoutes`；页面组件仍须存在于前端 |

关键链路：

- 页面：`src/pages/(base)/**`
- 元数据：`build/plugins/router.ts`（`ROUTE_META_PRESETS`）
- 生成：`pnpm gen-route` → `src/router/elegant/*`
- 动态挂载：`src/features/router/initRouter.ts` + `shared.ts`
- API：`src/service/api/route.ts` → `/auth/getRoutes`
- i18n：`src/locales/langs/{zh-cn,en-us}/route.ts`

### 2.3 可复用 vs 缺口

**可迁移（已有 CRUD / 列表雏形）：**

- 商品：`goods/{type,list,unit,quotation}` → `/master/goods/*`、`/master/pricing/quotations`
- 客户：`customer/{company,list,tag,sub-account,protocol}` → `/master/customer/*`、定价协议
- 采供：`purchase/{supplier,purchaser,rule}`、`storage/ware` → `/master/supply/*`
- 系统：`manage/{user,role,menu}` → `/system/*`
- 首页：`home` → `/dashboard/overview`（或 redirect）

**全新缺失（示例）：** 订单/售后/取货、采购计划与采购单、出入库与盘点查询、配送、财务、溯源、报表分析、系统运营支撑与打印工具等。

**演示菜单（建议 hide 或移除）：** `function`、`multi-menu`、`projects`、`about`。

## 3. 档 A：菜单骨架（3–6 人天）

机械工作为主，关键是前后端 `path` / `name` / `component` 对齐。

| 触点 | 位置 | 工作 |
| --- | --- | --- |
| 页面目录骨架 | `src/pages/(base)/` | 按文档建目录 + 占位 `index.tsx` |
| 生成路由 | `pnpm gen-route` | 刷新 elegant routes / imports / 类型 |
| 菜单元数据 | `build/plugins/router.ts` | `order`、图标、`hideInMenu`、`activeMenu` |
| 文案 | `src/locales/langs/*/route.ts` | 中英文 `route.*` |
| 动态路由 | `.env` + `initRouter.ts` | 切 `dynamic`，校验 `getRoutes` → `page.xxx` |
| 演示清理 | function 等 | 删或 `hideInMenu` |
| 联调 | 文档 §6 | 父节点补齐、隐藏详情页、角色裁剪 |

前提：后端已按文档写好菜单树。种子/权限码反复改时可再 **+1–2 人天**。  
后端种子 + Admin/演示角色绑定（本仓外）：约 **2–4 人天**。

---

## 4. 档 B：现有页 path 重组（+2–4 人天）

在 A 之上迁移已有模块，并回归列表/弹窗/按钮权限。

| 现路径 | 目标 path（文档） |
| --- | --- |
| `goods/*` | `/master/goods/*`；报价 → `/master/pricing` |
| `customer/*` | `/master/customer/*`；协议价 → pricing |
| `purchase/{supplier,purchaser,rule}` + `storage/ware` | `/master/supply/*` |
| `manage/*` | `/system/*`；其余系统叶子先占位 |
| `home` | `/dashboard/overview`（或保留 redirect） |

风险：`RouteKey`、i18n key、`imports.ts` 的 `page.xxx`、后端 `component`、已收藏 URL 会断。

---

## 5. 档 C：全量业务页（40–80+ 人天）

按「列表 + 基础 CRUD」粗估（不含复杂审批流/打印引擎），且假设 API 稳定：

| 中心 | 叶子数 | 粗估人天 |
| --- | --- | --- |
| 首页看板 | 2 | 3–6 |
| 订单中心 | 3 + 隐藏详情 | 8–15 |
| 采购中心 | 2 | 5–10 |
| 库存中心 | 10 | 12–25 |
| 配送中心 | 5 | 6–12 |
| 财务中心 | 4 | 6–12 |
| 溯源中心 | 3 | 4–8 |
| 报表分析 | 10 | 8–15（偏只读） |
| 基础资料 | ~13（含已有） | 已有可减半；新页约 3–6 |
| 系统管理 | ~11（含已有） | 已有可减半；新页约 4–8 |

合计约 **40–80+ 前端人天**。这是「按菜单把产品做完」，不要与「更新菜单结构」混谈。

---

## 6. 建议落地顺序

1. **后端先定种子**：冻结文档中的 `path` / `name` / 权限码。
2. **前端 A**：占位树 + 切 `dynamic` + 过文档 §6 联调清单。
3. **前端 B**：只迁已有 CRUD，保证生产可用模块 path 正确。
4. **按优先级填 C**：订单 → 采购 → 库存 → 配送 → 财务…（与文档 §4 侧栏排序一致）。
